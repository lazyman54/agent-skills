# 自测 Reviewer 复核清单

> reviewer 拿到自测文档（`docs/self-test/{branch}_{date}.md` 或 `docs/self-test/{branch}_{date}/`）后逐项复核。**任意 A 类红线 fail = 整份退回**。

## 复核原则

1. **看证据，不看声明**：开发者写"已通过"不算数，必须能在主文档"实际结果"小节看到真实输出
2. **抽查 + 红线**：A 类（真实性）必须 100% 抽查通过；其他类按比例抽查
3. **退回比放行容易**：宁严不放——自测放行的代价远大于多 review 半小时

---

## A. 真实性（红线项）

### A.1 文档存在 + 路径合规

- [ ] 文档路径符合两种形态之一：
  - 单场景：`docs/self-test/{branch_slug}_{date}.md`
  - 多场景（≥2）：`docs/self-test/{branch_slug}_{date}/README.md` + 每场景一份 `{scene}.md`
- [ ] feature 触及 ≥ 2 场景却用单文件且总行数 > 1000 → 退回，要求拆目录式
- [ ] feature 仅触及 1 场景却建了目录 + 单个 scene.md → 退回,要求合并回单文件
- [ ] 多场景形态下，README 只放跨场景共用部分（概述/口径/索引/汇总/Observability/冲突清单），分支细节归各场景 .md
- [ ] `branch_slug` 与 `git branch --show-current` 对得上（仅 `/` 转 `-`）
- [ ] `date` 是开测日期，非未来日期

### A.2 4 类输入文档已收集且冲突已确认

- [ ] 概述章节列出 PRD / 技术方案 / use-case / 测试用例 4 类文档链接
- [ ] **"已确认的文档冲突"小节有内容**（写"无"也算明确表态；但 4 类文档间不可能 100% 一致，写"无"是可疑信号）
- [ ] 每条冲突均已 user 确认采信哪份文档

### A.3 实际证据非"未执行"占位

- [ ] 声明 PASS 的分支：`3.X.4 实际结果` 三个子小节（数据/日志/返回）均有真实输出，**不是 ⏸ 未执行 占位**
- [ ] DB 输出含真实主键、时间戳，不是虚构占位
- [ ] 日志含 trace_id、时间戳，trace_id 在 RPC 响应或 SQL 中能互相印证
- [ ] 抽查 1-2 个 PASS 分支，从证据独立推得 PASS 结论

### A.3.1 分支结构合规（防"测试前检查"段缺失）

- [ ] 每个分支用五段式编号 `3.X.1 测试前检查` / `3.X.2 构造入参` / `3.X.3 预期结果` / `3.X.4 实际结果` / `3.X.5 判定`（需前置改造的分支前面再加 `3.X.1 前置改造`，其他段顺延）
- [ ] `3.X.1 测试前检查` 段按"三步检查法"组织：前置 SELECT → 数据准备 → 准备后 SELECT，三步缺一退回
- [ ] 分支 h4 标题都带 `3.X.Y` 编号——若直接写 `#### 测试前检查` 会导致 GFM 锚点冲突，目录跳转失败

### A.4 时间戳一致性

- [ ] 实际结果中的日志时间戳落在自测人声明的"自测日期"窗口
- [ ] 不同分支的时间戳合理递增（除非显式标记重跑）

> A 类任意一条 fail → ❌ REJECTED，开发者补全证据后重新提交。

---

## B. 完整性（防漏覆盖）

### B.1 范围声明完整

- [ ] "本 feature 涉及的场景"清单与 `git diff main...HEAD --stat` 推导一致
- [ ] **没有列被测系统全部场景** — 应只覆盖本 feature 触及（写全部 = 漏掉了"feature 范围"原则）
- [ ] 现有用例覆盖度评估有内容

### B.2 git diff 触及的代码都有覆盖

```bash
git diff main...HEAD --stat
```

- [ ] 每个 diff 涉及的核心函数 / 状态转移在某个分支被覆盖
- [ ] 没有"顺手改动" — 每行 diff 都能追溯到本 feature 的某条 AC

### B.3 异常路径覆盖

- [ ] 主路径（happy path）有 PASS 证据
- [ ] 至少 1 条异常分支被覆盖（mutex / 上游异常 / 鉴权失败 / 入参非法 等择一）
- [ ] 跳过的异常分支显式标 SKIP + 原因 + issue 链接

---

## C. 三维度证据齐全（防"只看日志"）

### C.1 关键分支三类证据齐全

主路径 + 关键边界分支必须**同时**有：

- [ ] 数据小节有真实 DB 输出
- [ ] 日志小节有真实日志
- [ ] 返回小节有真实 RPC 响应 / Kafka 消息

仅单一来源（"日志正常所以通过"）的关键分支：❌ 退回。

### C.2 状态机变更专项

涉及状态转移的分支：
- [ ] 数据小节有 before / after DB 状态对比
- [ ] 有完备性单测（参考项目 `.claude/rules/` 下相关测试规范，如有）

### C.3 Kafka / 监控

- [ ] 涉及 Kafka 投递的分支：有 producer 日志或 `dispatched_at` 类字段写入证据（如 observability 是已知短板，看是否在 O 节列入改进）
- [ ] 涉及监控/告警：有 metric 上报记录或文档明确说明缺埋点

---

## D. 风险声明（防带病上线）

### D.1 FAIL / PARTIAL / SKIP 项处理

- [ ] FAIL 项：明确修复 commit 链接 + 重跑证据
- [ ] PARTIAL 项：明确缺什么 + 是否阻塞验收
- [ ] SKIP 项：原因合理 + issue 链接 + 时间承诺

### D.2 发布声明明确

- [ ] "整体验收结论"末尾发布声明**不含"应该没问题"等含糊表述**
- [ ] 若带病上线：写明未解决项 + 风险评估 + 应急方案 + 已周知 leader/QA

---

## E. 项目规范一致

按项目下 `.claude/rules/` 与 `.specify/memory/constitution.md`（如有）的规则文件检查：

- [ ] 状态机变更有完备性单测（按项目测试规范）
- [ ] 错误码符合项目号段（按项目错误码规范，号段从项目规则文件读取）
- [ ] 监控 / 日志 关键字符合项目规范

## 附录：常见的"看起来通过实际有问题"反模式

reviewer 重点警惕：

| 模式 | 识别信号 | 应对 |
|------|---------|------|
| 声明驱动 | 全是"已通过"、"符合预期"，没有真实输出 | 退回 A.3 |
| 只看日志 | 关键分支仅有日志证据，无 DB / 返回对照 | 退回 C.1 |
| 截图代替 | 实际证据是截图链接而非文本 | 退回，要求文本输出 |
| 裁剪证据 | 日志只贴成功的几行，前后异常被截掉 | 要求贴完整时间窗口 |
| 自我修正 | 发现 fail 后改测试代码 / SQL 让结果通过 | 严肃退回，要求改业务代码 |
| 打包跳过 | 大量 SKIP 集中在某场景，理由都是"以后" | 要求逐个解释或转 PARTIAL |
| 范围扩张 | 写了被测系统全部场景而不是 feature 范围 | 退回 B.1 |
| 范围缩水 | git diff 触及的函数没在文档中 | 退回 B.2 |
| 含糊声明 | "应该没问题" / "估计 OK" | 要求明确 D.2 |
